Sketch – icon management for multiple libraries

Let’s imagine a situation when the same set of icons is used in different design systems. We have a brand A and B with different color profiles and the icons have to inherit the color after particular profile. We have a one of the icons sets ready for the brand A with the primary and secondary color: What we want to achieve is:

  • Possibility of global changes in the icons
  • Quick new color to apply for the icon set
  • Possibility of global and local changes in the border styles (for example from curved ends to straight ones

Of course, we are working on global color settings and have the colors defined as variables:

Approach 1

Then for the Brand B I will have only to set the new value of the primary color. But (there is always a but) in Sketch as far as I know it is not possible to have the root library with basic settings that we can propagate to local libraries and keep them synchronized. Sketch works well with one library and one design system but what if as in my case we have three different brands using the same but customized design system? The only way is to clone the existing library for Brand A and use the new one for Brand B. The synchronization is gone. So maybe the better idea would be to use only 1 library for all brands and just override default values with the local ones? I have tried this and after a while I noticed it is a bad idea. The number of bugs that appear after adjusting color values, text and layer styles is huge. The only option was to roll back and have just one stand-alone library for one brand.

Approach 2

The goal is to clone the library and with minimum effort adapt it to the new brand globally applying new colors, text and layer styles.

First step is changing the border style structure. This will help us quicker apply global changes in the icons. My initial idea was to group border styles with the color as a root of the structure and then grouped by the thickness starting from 1 for 16x16 px icon size, 1,5 for 24x24 and 2 for 40x40px. We can call it of course a different way for example thin, medium, thick because not in all designs systems we create the basic border with might be 1px. What I noticed with this solution was that I have the repetitive pattern with border width. It also has an impact on the speed of applying changes to the icons as we have separately chosen the color to change and expand many windows in Sketch to apply the change: So, the next step was to organize the structure according to the border thickness with border colors in the sublevels. Something like this: Thanks to this we have immediate access to all palette color, so the global changes are less painful. This is convenient in the phase of corrections and bug fixing. I struggled a lot in the inconsistency in the styled used in the icons but also in the next step which was creating a kind of master for the set of the same icons in different colors. Default icon is editable while all next variants use only border styles as overrides. Thanks to this if the icon needs to be changed only master icon is edited and the rest have the local border styles applied from quick access palette: I thought even on going one step further and keeping the master icons in the separate library that would be a global library for my all-design systems, but then again, I encountered the problem and question how in the Brand A Design System library manipulate in the global way with for example border thickness. It turns out that the solution which should help us to keep the flexible and easy to adjust system works against us. So, I decided to keep the solution with local icon masters and full access to all variables for brand Design System. Maybe in future the solution of the global icon library wins but at this moment when brands are still being tested and developed, I want to have an option of full access to local variables for one brand.

To sum up. The research I have done is about the icons but in my case touches also other elements of the Design System serving several brands because I encountered similar problem when thinking about for example the buttons (different fonts but the same global patterns for states, global paddings, and margins but local Design System icons using local colors).
